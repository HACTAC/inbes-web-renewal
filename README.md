# INBES株式会社 Webサイトリニューアル

## このREADMEの役割

本書は、INBES株式会社コーポレートサイトの現行方針、実装状況、未確定事項を共有するための進行メモである。

過去の検討過程ではなく、現在採用している判断のみを記載する。方針を変更した場合は本書を更新し、古い案は残さない。

## 現在の結論

INBESのWebサイトは、次の2つの目的を持つ。

1. INBESが企画・販売する自社製品を探すユーザーの入口
2. 日本市場向けの商品化を相談したい法人の入口

トップページは会社説明を行うページではなく、訪問目的をこの2系統へ分ける入口とする。

- 自社製品
- 商品化支援

PCでは左右2分割、モバイルでは上下2分割とし、原則としてファーストビュー内で選択できる構成にする。

## 正本と参照ファイル

### 実装の正本

現在の正本はAstro実装である。

- ページ: `src/pages/`
- 共通レイアウト: `src/layouts/`
- コンポーネント: `src/components/`
- データ: `src/data/`
- 共通スタイル: `src/styles/`

今後の構成・コピー・デザイン変更は、最終的にAstro実装へ反映する。

### 旧HTMLワイヤーフレーム

`wireframes/top-v2/` と `wireframes/top-v3/` は、必要な製品・実績・ポリシー情報をAstroへ移行したため削除した。

今後はAstro実装をワイヤー兼実装の正本として更新し、HTMLワイヤーフレームを二重管理しない。

### デザインシステム

デザインシステムは `design-system/` で検討する。

Brand OSをWeb用のFoundation、Component、Patternへ展開したレビュー用ページを `design-system/site/` に置く。承認前の内容はAstro実装やワイヤーフレームへ反映しない。

ブランド表現の参照元は `HACTAC/INBES-BrandOS` とする。ローカル参照データは `references/INBES-BrandOS/` に置き、Git管理対象には含めない。

- デザインシステム確認URL: `https://hactac.github.io/inbes-web-renewal/design-system/`
- 状態区分: `Confirmed` / `Proposed` / `Review`

## ブランドとサイトの立ち位置

INBESは、自社製品の企画・販売とOEM/ODMで培った経験をもとに、日本市場に合わせた商品化を支援する会社として表現する。

商品を製造するだけではなく、仕様整理、ローカライズ、量産、輸入、パッケージ、取扱説明書、販売準備までを必要に応じて支援する。

中心となる考え方:

- 日本で売るための商品づくり
- 世界にある技術を、日本の暮らしへ
- 作ることだけでなく、販売できる状態まで整える
- 強く売り込まず、相談しながら一緒に整理する

## 想定ユーザー

### 自社製品側

- INBES製品を探しているユーザー
- 製品仕様や販売ページを探しているユーザー
- 購入済み製品について問い合わせたいユーザー

### 商品化支援側

- 国内に販路や顧客接点を持つ企業
- 新しい商品を扱いたい企業
- 海外製品を日本市場向けに販売したい企業
- OEM/ODMや量産、輸入を外部へ相談したい企業
- パッケージや取扱説明書など販売準備まで依頼したい企業

## サイト構成

### トップ

役割: 訪問者を「自社製品」と「商品化支援」へ分ける。

- 自社製品 → `/products/`
- 商品化支援 → `/services/`
- 補助導線として会社概要と問い合わせを配置する

### 自社製品側

#### 製品一覧 `/products/`

自社で企画・販売する製品を、商品単位で掲載する。詳細情報が既存LPや公式ページにある場合は外部リンクで案内する。

#### 製品サポート `/contact/support/`

自社製品に関する問い合わせ窓口。一般的な問い合わせはフォームと電話で受け付ける。

自社製品側ナビゲーション:

1. 製品一覧
2. サポート

製品一覧ページ内にサポート枠は置かず、ヘッダーの「サポート」から専用フォームへ直接案内する。

商品化支援側への相互リンクとして「商品化をご相談の方」を表示する。

### 商品化支援側

#### 商品化支援 `/services/`

INBESが何を支援するか、どのような考え方で商品化するかを伝える。

主な内容:

- 商品化で起きやすい課題
- ローカライズ開発
- オリジナル開発
- 主な取扱分野
- 進め方への導線
- 開発実績カルーセルと相談導線

ローカライズ開発を主軸とし、オリジナル開発は必要に応じた選択肢として扱う。ページの順序は「課題 → 商品化の考え方 → 取扱分野 → 進め方 → 開発実績・相談」とする。

ローカライズ開発では、国内仕様、表示・ラベル、取扱説明書、許認可・国内試験、パッケージ、品質・輸入条件を日本市場に合わせて整えることを簡潔に示す。

ページ末尾は、開発実績のカルーセルと次の2つの行動を一つのセクションに統合する。

- 商品化について相談する
- 開発実績を見る

#### 進め方 `/process/`

相談から販売開始までの流れを伝える。

- 作りたい商品がある
- こんな商品を探したい
- 商品化プラン
- 試作・確認
- 量産・輸入
- 納品
- 継続サポート
- 商品まわりのデザイン

#### 開発実績 `/works/`

「どんな商品を、どこまで支援したか」が一覧で分かる構成とする。

掲載項目:

- 商品画像
- 商品名
- カテゴリ
- 支援内容タグ
- 一文紹介
- クライアント
- 公式サイトへのリンク（公開案件のみ）

ページ末尾は、次の2つの行動を並列で案内する。

- 商品化について相談する
- 商品化支援を見る

「進め方」はヘッダーナビゲーションと商品化支援ページから案内し、開発実績ページ末尾の選択肢は増やさない。

#### 会社概要 `/company/`

Message、About、Profileの3層で構成する。

- Message: INBESの考え方
- About: INBESが行う事業
- Profile: 会社情報

#### 商品化の相談 `/contact/business/`

仕様や進め方が固まっていない段階でも相談できる窓口とする。

商品化支援側ナビゲーション:

1. 商品化支援
2. 進め方
3. 開発実績
4. 会社概要
5. お問い合わせ

自社製品側への相互リンクとして「製品をお探しの方」を表示する。

### 問い合わせ入口 `/contact/`

次の2つの窓口を選択するページとする。

- 商品化のご相談
- 製品サポート

## 技術構成

- フレームワーク: Astro
- CSS: Tailwind CSS
- JavaScript: 必要最小限
- ソース管理: GitHub
- 本番公開先: mixhost
- CMS: 初期公開では使用しない
- 将来拡張: microCMS

VercelやGitHub Pagesを本番環境には使用しない。GitHub Pagesは確認用としてのみ利用する。

### ワイヤー確認URL

- GitHub Pages: `https://hactac.github.io/inbes-web-renewal/`
- `main`へプッシュすると、GitHub ActionsがAstroをビルドして確認サイトを更新する
- GitHub Pagesではフォーム送信を確認できない。フォーム送信はmixhost上のPHP実装後に検証する

製品、実績、サービスなどの情報は、将来microCMSへ置き換えられるよう、データと表示コンポーネントを分離する。

## 問い合わせフォーム

フォームUIはAstroで作成し、mixhost上のPHPで送信処理を行う方針とする。

フォームは次の2種類に分ける。

### 商品化の相談

- 会社名 任意
- お名前
- メールアドレス
- 電話番号 任意
- ご相談内容
- 個人情報の取り扱いへの同意

### 製品サポート

- お名前
- メールアドレス
- 電話番号 任意
- 製品名、購入時期、問い合わせ内容
- 個人情報の取り扱いへの同意

実装時には、SMTP送信、自動返信、送信ログ、エラー表示、スパム対策、管理者宛メールの到達確認を行う。

## デザイン方針

- 情報量ではなく、構造と優先順位を見せる
- 見出しだけでもページ内容が理解できるようにする
- 余白と写真を活かす
- CTAは押し売り感を抑え、相談しやすくする
- 商品化支援側と自社製品側の違いを、情報設計と視覚の両方で伝える
- 過度なグラデーションや未来感だけの表現を避ける
- デザイン会社のサービス紹介に見せない
