# INBES株式会社 ウェブリニューアル 制作方針メモ

## 結論

INBES株式会社のウェブサイトは、単なる会社案内ではなく「販売ルートを持つ企業のために、次のエレクトロニクス製品を企画から量産まで支援するODMパートナー」であることが短時間で伝わる営業・信頼形成の拠点として設計する。

初期方針は、派手な演出や情報量の多さではなく、必要な情報を簡潔に分かりやすく伝えることを優先する。ブランド表現は、見た目の装飾ではなく「誰のどんな課題を前に進める会社なのか」を明快に示すことに宿らせる。

現時点の中心テーマは、自社製品販売ではなく、エレクトロニクス領域、特にコンシューマ系のジェネリック家電・周辺機器のODMである。自社製品は信頼の補助情報として扱い、サイト全体の主役は受託開発・ODMに寄せる。

## 現行マスターワイヤーフレームの優先方針

このセクションを、以降に残る検討メモより優先する。

### サイト構成

グローバルナビゲーションは以下の順番を基本とする。

1. 自社製品
2. 開発支援
3. 開発実績
4. 会社概要
5. お問い合わせ

お知らせは初期公開では必須にしない。更新体制が見えた段階で追加を検討する。

### ページごとの役割

- トップ: INBESの立ち位置を短く伝え、各下層ページへ送る入口。
- 自社製品: 既存サイト掲載商品を整理し、詳細は既存LPや外部ページに送る。
- 開発支援: 主力ページ。ローカライズ開発を軸に、必要に応じてゼロ開発、進め方、デザインサポートを案内する。
- 開発実績: 受託開発・共同開発の実績を、製品名、クライアント、関与範囲、外部リンクとともに整理する。
- 会社概要: MESSAGE、PHILOSOPHY、PROFILEの順で信頼形成に必要な情報を整理する。
- お問い合わせ: よくある質問で不安を減らしつつ、フォームは簡潔にする。

### トップページの現在の見せ方

トップページは、各下層ページへ送る入口として設計する。LPのようにトップだけで説明を完結させず、訪問者が次に見るべきページへ自然に進める構成にする。

HERO右側は開発支援の説明カードではなく、既存ユーザー・一般ユーザー向けの「製品をお探しの方へ」導線として扱う。サムネイル付きの小さなフローティングバナーにし、自社製品一覧へ送る。

現時点の中段以降の順序は以下を基本とする。

1. ISSUE: 次の商品化で起きやすい課題を提示し、開発支援へつなげる
2. SERVICES: 開発支援の入口として、見出し、本文、写真、ボタンのみで簡潔に見せる。ボタンは「開発支援」と「開発実績」の2つにする
3. PRODUCTS: 既存ユーザー・一般ユーザー向けの入口として、横長バナー、カテゴリタグ、一覧ボタンで自社製品へつなげる
4. COMPANY: 自社製品販売とOEM/ODMの経験、社訓、会社概要への導線だけに絞り、詳細な基本情報は会社概要ページへ送る

ナビゲーション上の主要導線としては、開発支援、開発実績、自社製品の順で見せる。自社製品は信頼補強として扱い、ODM・開発支援より前に出しすぎない。

開発支援ページと開発実績ページの下部には相互リンクを設ける。開発支援を見た人が実績へ進み、実績を見た人が相談できる内容へ戻れるようにする。

### 開発支援ページの考え方

開発支援ページは、単に「できること」を並べるページではなく、INBESに相談した場合の進め方を理解してもらうページとする。

現時点では、海外既存製品を日本向け・販売先向けに整える「ローカライズ開発」が主軸である。ゼロからの新規開発は実績があるものの、主役として強く出しすぎず、必要な場合に検討できる開発方法として見せる。

次回調整する論点:

- `SCOPE` と `ZERO DEVELOPMENT` は別事業のように見えやすいため、英字ラベルと見出しを整理する
- 「開発アプローチ」の中に、2つの進め方がある構造として見せる
- ローカライズ開発を主軸、ゼロ開発を補助的な選択肢として強弱をつける
- 見出しは文章感を減らし、短くプロらしい表現へ寄せる

表現候補:

- 親見出し: 開発アプローチ
- 主軸: ローカライズ開発
- 補助: ゼロからの開発
- 英字ラベル: `Approach 01` / `Approach 02` または `Primary` / `Optional`

`Scope` は「対応範囲」の意味が強く、現在の意図である「開発の進め方」と少しずれる。公開前には `Approach 01` などのラベルへ変更する方がよい。

### 先方確認と素材の前提

ワイヤーフレームの方向性は先方確認中。現時点では、大枠の構成はこの方向で進める前提とする。

素材については、商品系の一部に不足があるものの、基本的には揃う前提で考える。不足するイメージ素材は、有料素材またはAI生成を使って補う。代表写真は掲載を検討するが、社屋写真は用意しない前提とする。

問い合わせ先、フォーム送信先は既存サイトの情報を踏襲する。開発実績については、各案件でどの範囲まで支援したかを確認する必要があるため、ヒアリング項目を作って整理する。

### 制作方式

制作方式は、Astro + microCMS に挑戦する方針とする。

WordPressは既存運用として現実的な選択肢だが、今回のINBESサイトでは、静的生成による表示速度、管理画面の入力項目設計、将来的な拡張性を考え、Astro + microCMS を第一候補として進める。

実装方針は、現在のHTMLワイヤーフレームをもとに、まずAstro側でページ構造とコンポーネントを整理し、その後に自社製品、開発実績、必要に応じてお知らせ等をmicroCMS管理に切り出す。

## 中心コピー

御社の商流に乗る、次のエレクトロニクス製品を。

このコピーを、現時点のトップページ最有力案とする。

狙い:

- 販路を持つ企業に直接刺さる
- 単なる製造委託ではなく、事業成長の文脈で伝わる
- 「作れるもの」ではなく「売り場に乗せられる商品」に焦点が合う
- INBESの役割が、製造会社ではなく商品開発パートナーとして見える

補助コピーの方向:

販売ルートを持つ企業に向けて、コンシューマ向けエレクトロニクス製品の企画・仕様検討・ODM開発・量産を支援します。

## 編集方針

情報を多く盛り込むのではなく、必要な情報を簡潔に分かりやすく伝える。

重視すること:

- 一文で何の会社か分かる
- 見出しだけでページの構造が追える
- 専門用語を使う場合は、前後の文脈で意味が分かる
- 説明しすぎず、問い合わせにつながる余白を残す
- 実績や対応範囲は、羅列ではなく判断材料として編集する
- デザインは装飾ではなく、情報の優先順位を整えるために使う

避けること:

- ODM、OEM、量産、海外工場などの専門語を並べるだけのページ
- 理念や社訓を長文で読ませる構成
- 自社製品を前面に出しすぎてメーカーサイトに見える構成
- できることを広く書きすぎて、何に強い会社か分からなくなる表現
- 最新感を出すためだけの抽象的なビジュアル表現

## ターゲット仮説

主な対象は、国内に販売ルートを持つが、商品開発力や製品化のスピードに課題を持つ企業。

具体像:

- 量販店、通販、EC、専門店、代理店などの商流を持っている
- 既存顧客や売り場に提案できる新商品を探している
- しかし、自社内の商品開発体制が弱くなっている
- 海外工場との調整、品質管理、仕様決定に不安がある
- 完全な新規発明ではなく、既存市場にある製品カテゴリをベースに、自社の販路に合う商品を作りたい

このターゲットに対して、INBESは「御社の販売力を、次の商品開発につなげる会社」として見せる。

## トップページの課題提示

課題提示は、以下の3つを基本にする。

1. 売り場に提案する新商品が足りない
   - 既存の商流や顧客接点を活かしたいが、次に扱う商品企画が見つからない。

2. 変化するニーズを、製品仕様にできていない
   - AI、アプリ連携、ネットワーク対応など、商品に求められる要素は変化している。売り場に乗せられる仕様へ落とし込むには、企画と製造をつなぐ視点が必要である。

3. 製品化の実務を、社内だけで抱えきれない
   - 仕様整理、試作、品質確認、工場調整までを社内で担うには、時間も人材も必要である。必要な部分を外部化することで、商流と販売に集中できる。

この3つにより、INBESを単なる製造委託先ではなく、商品開発機能の一部を外部化できるパートナーとして見せる。

## 生産体制とデザイン連携

INBESの特徴として、製造・量産だけでなく、デザイン会社との連携により「売り場に出すための準備」まで支援できる点を打ち出す。

伝えるべきこと:

- 製品仕様、製造、量産だけでなく、パッケージやWebサイトなどの制作部分も相談できる
- デザイン会社と連携しているため、販売時の見せ方を一定以上の水準で整えられる
- 商品そのものと、商品を売るための表現を分断せずに考えられる
- 販売ルートを持つ企業にとって、製品化後の展開まで見通しやすくなる

表現上の注意:

- デザイン会社の存在を前面に出しすぎると、INBESの主役感が弱まる
- 「提携デザイン会社」という言い方は、実態に合わせて慎重に使う
- サイト上では「デザイン会社との連携により、パッケージやWebサイトなども含めて相談できる」程度から始める
- 具体的な社名や担当範囲を出すかは、先方確認後に決める

安心と安全を支える製品を通じて、社会に信頼と価値を届ける。

### 社訓

- 誠実
- 挑戦
- 責任

### 信念

お客様に誠実に向き合い、新しい価値づくりに挑戦し、製品に最後まで責任を持つ企業であり続ける。

### 財務理念

無理な拡大や短期的な利益追求に偏らず、堅実な財務基盤のもとで持続的に価値を提供する。健全な資金管理と適切な投資判断により、お客様・取引先・社会に対して責任ある事業運営を行う。

### 行動指針

- お客様目線を大切にする
- ものづくりに誠実である
- 変化を恐れず挑戦する
- 信頼される仕事を積み重ねる

### 代表メッセージの要点

- 防犯機器をはじめとする自社製品の販売実績がある
- 各種電気製品の受託開発、OEM/ODMを主軸として打ち出す
- 単に製品を提供するのではなく、お客様とともに市場に出せる製品をつくる
- 安心・安全、品質への責任、現実的かつ柔軟なものづくりを重視する
- ものづくりのパートナーとして価値提供を続ける

## ポジショニングの変更

### 変更前の見え方

自社製品を持つ防犯・電気製品メーカー。

### 変更後の見え方

コンシューマ向けエレクトロニクス製品を、企画・仕様検討・試作・量産まで支援するODM会社。

### 打ち出しの優先順位

1. ODM・受託開発
2. コンシューマ系エレクトロニクスへの対応力
3. 企画から量産までの実務対応
4. 品質・コスト・納期のバランス提案
5. 自社製品の実績

### 自社製品の扱い

自社製品はメイン商材として大きく打ち出すのではなく、以下の役割で扱う。

- 製品化経験の証明
- 品質管理や量産経験の補強
- 防犯・監視領域への対応実績
- ODM相談時の信頼材料

トップページでは自社製品を前面に出しすぎず、「自社製品で培った製品化ノウハウを、ODM開発に活かす」という位置づけにする。

## 開発支援ページの基本方針

### キャッチコピー

企画を、量産できる製品へ。

補助コピー候補:

- コンシューマ向けエレクトロニクス製品のODM開発を、企画から量産まで支援します。
- ジェネリック家電・周辺機器の製品化を、現実的な仕様と製造体制で支えます。
- アイデア段階から量産を見据え、品質・コスト・納期のバランスを設計します。

### 伝えるべき価値

INBESの受託開発は、単なる製造委託ではなく、構想・仕様検討・試作・評価・量産・海外工場調整までを含む「現実的な製品化支援」として見せる。

特に、完全な新規発明品よりも、既存市場にある製品カテゴリをベースに、仕様変更、機能追加、デザイン調整、パッケージ化、販路に合わせた製品化を行うODMパートナーとして伝える。

### 対応内容

- 製品企画・仕様検討
- OEM/ODM開発
- 既存製品のカスタマイズ
- 試作・評価
- 量産立ち上げ
- 海外工場との調整
- 部材・構成提案

### 対応分野

- コンシューマ向けエレクトロニクス製品
- ジェネリック家電
- 生活家電・小型電気製品
- 電源機器・周辺製品
- カー用品・車載アクセサリー
- 防犯・監視関連機器
- 各種一般電気製品

### 想定顧客

- 家電・雑貨・カー用品などを扱う販売会社
- EC、通販、量販店向けに独自商品を作りたい企業
- 既存製品をベースに自社ブランド商品を展開したい企業
- 海外製品の仕様調整や国内向け展開を検討している企業
- 企画はあるが、製造・量産・工場調整の体制がない企業

### 補完が必要な情報

- 開発対応できる範囲の線引き
- 設計、基板、筐体、ファームウェア、アプリ、認証対応の可否
- 国内対応と海外協力工場の役割分担
- 最小ロット、量産規模、概算期間
- 得意な製品カテゴリ
- 過去実績、事例、写真、図面掲載の可否
- 品質管理、検査体制、保証範囲
- 相談から納品までの標準フロー
- 問い合わせ時に必要な情報
- 「ジェネリック家電」と表現してよいか、別表現にするか
- 国内販売時の法規制、PSE、技適、電波法、各種認証への関与範囲

- 実績、事例、思想を蓄積できるメディア基盤にする
- 将来的な採用、協業、地域案件にも展開できる余白を持たせる

## 重要な設計仮説

### 1. 「何をしている会社か」より「どんな課題を解く会社か」を前面に出す

企業サイトでは事業内容を並べがちだが、訪問者が知りたいのはサービス名ではなく、自分の課題と接続できるかである。

そのためトップでは、業務内容の羅列よりも、「作りたい製品はあるが、仕様・工場・量産・品質で止まっている」という課題に対して、INBESがどこまで伴走できるかを先に見せる。

### 2. 実績は作品集ではなく、判断材料として見せる

単に制作物を並べるだけでは、発注者には判断しづらい。背景、課題、提案、成果、役割を短く整理し、INBESの思考と実行力が伝わる形式にする。

### 3. 会社の思想を「代表挨拶」に閉じ込めない

理念や考え方は、抽象的な文章だけでは伝わりにくい。サービス説明、事例、制作プロセス、問い合わせ導線の各所に分散して表現する。

### 4. 小規模でも更新し続けられる構成にする

初期から大規模なCMS運用を前提にしすぎない。更新対象を絞り、実績・お知らせ・コラムなどの運用負荷を現実的に設計する。

### トップ

役割:
INBESを「コンシューマ向けエレクトロニクスのODMパートナー」として即座に理解してもらうページ。

訪問者の疑問:
- 何の会社か
- どんな製品づくりを相談できるか
- どの段階から相談できるか
- 信頼できる会社か
- 問い合わせる価値があるか

載せる内容:

役割:

## 技術方針

第一候補を、Astro + microCMS + Cloudflare Pages 構成へ変更する。

理由は、今回のサイトが「情報を多く盛り込む」よりも「必要な情報を簡潔に、精度高く伝える」ことを重視するためである。主要ページは制作側がコードで品質を管理し、更新が必要な部分だけCMS化することで、デザイン品質と運用性を両立する。

WordPressは依然として有力な代替案だが、今回は今後の制作資産としても活かせる構成に挑戦する。

- CMS: microCMS

スタッフには「ページ編集」ではなく「決められた項目への入力」を任せる。表示デザインや構成はAstro側で管理する。

製品化の実績、自社製品のCMS項目案:

個別詳細本文は初期段階では持たない。必要になった場合のみ追加する。

### 問い合わせフォーム方針

問い合わせフォームは、外部フォームに遷移させず、サイト内に設置する。

構成:

- AstroでフォームUIを作成
- Cloudflare Pages Functionsで送信処理
- Cloudflare Turnstileでスパム対策
- Resendで管理者宛メールと自動返信メールを送信
- 必要に応じてGoogleスプレッドシートなどに控えを保存

フォーム項目の初期案:

- 会社名
- お名前
- メールアドレス
- 電話番号 任意
- ご相談内容
- 想定している製品カテゴリ
- 販売先・商流 任意
- 希望時期 任意
- 相談内容の詳細
- NDAが必要か
- 個人情報の取り扱いへの同意

注意点:

- 問い合わせはメール送信だけで終わらせず、控えを残す
- 個人情報をmicroCMSに保存する場合は管理責任が増えるため、初期は避ける
- 送信完了ページ、エラー表示、自動返信を用意する
- スタッフが問い合わせ内容を見落とさない運用を決める

### 後悔しないための境界

- 全ページをCMS化しない
- スタッフがレイアウトやデザインを変更できる状態にしない
- microCMSの入力項目を細かく設計しすぎない
- 問い合わせフォームのスパム対策を後回しにしない
- 公開前プレビューと確認フローを用意する
- 主要ページの文言変更は制作側で管理する

### 候補A: WordPress

推奨度: 中

メリット:
- 更新しやすい
- 既存運用に乗せやすい

デメリット:
- 保守、セキュリティ、プラグイン管理が必要
- 表示速度や構造が雑になりやすい
- テーマやプラグインに依存しすぎると長期保守が弱くなる

採用する場合の方針:
- ブロックエディタを活かす
- 必要最小限のプラグインに絞る
- 製品、実績、お知らせは投稿タイプを分ける
- デザインの自由度が必要なページはテンプレート化する
- SEO、フォーム、セキュリティ、バックアップは初期から設計する

推奨度: 中

メリット:
- 表示が速い
- 保守が軽い
- セキュリティリスクが低い
- 会社案内中心で更新頻度が低い場合は合理的

デメリット:
- 非エンジニア更新には工夫が必要

- 製品や実績を増やすたびに制作側対応になりやすい

- 管理画面が不要
- 公開後の保守コストを最小化したい
- 開発実績や自社製品を頻繁に増やさない

### 候補C: ヘッドレスCMS + 静的生成

推奨度: 高

メリット:
- 表示速度と更新性を両立しやすい
- 将来的な拡張に強い
- microCMSなどを使えば日本語管理画面とAPI連携を使える

デメリット:
- 初期設計がやや複雑
- 小規模サイトには過剰な場合がある
- ホスティング、CMS、ビルドの責任範囲が分かれる

向いている条件:
- 表示速度を強く重視する
- 将来的に複数サイトやアプリへ同じ情報を配信したい
- 製品DB、事例DBとして情報を再利用したい
- 制作側が継続的に技術保守できる

### 候補D: Astro + Markdown/MDX

推奨度: 中

メリット:
- 企業サイト、サービスサイト、記事コンテンツと相性がよい
- 静的生成で高速にしやすい
- Markdown/MDXで実績やコラムを構造化できる

デメリット:
- クライアント自身が管理画面で更新する用途には弱い
- 更新フローをGitまたは別CMSと組み合わせる必要がある

向いている条件:
- 制作側が更新を引き受ける
- 高速で軽いサイトにしたい
- WordPressの保守を避けたい

### 候補E: Next.js

推奨度: 低から中

メリット:
- 複雑なWebアプリ、会員機能、検索、フォーム、外部API連携に強い
- 将来的な機能拡張に強い

デメリット:
- 今回の一般的な企業サイトにはやや過剰
- 保守にはフロントエンド開発の継続体制が必要

向いている条件:
- 製品検索、会員向け資料、見積もりシミュレーターなどを本格的に作る
- 将来的にサイトを業務システム寄りに拡張する
